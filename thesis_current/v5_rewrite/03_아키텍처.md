# 제Ⅲ장 Triple-Hybrid RAG 아키텍처 (v5 재작성)

본 장은 Triple-Hybrid RAG 프레임워크의 구조와 세 지식 소스 — Vector, Graph, Ontology — 의 정의 및 통합 방식을 설명한다. Ch.4의 R-DWA와 Ch.5의 L-DWA는 모두 본 아키텍처 위에서 작동하며, 본 장은 이들이 **공유하는** 파이프라인의 정의이다.

## 1절 전체 파이프라인

사용자 자연어 질의 `q`는 다음 7단계를 거쳐 답변 `a`로 변환된다.

```
(1) q
(2)   → QueryAnalyzer → intent (entities, relations, constraints, density)
(3)   → DWA (R-DWA or L-DWA) → (α, β, γ) ∈ Δ³
(4)   → {VectorStore.search, GraphStore.search, OntologyStore.search} → (V, G, O)
(5)   → merge_contexts(V, G, O; α, β, γ) → context string
(6)   → LLM(prompt + context) → answer a
(7) a
```

![그림 3-1. Triple-Hybrid RAG 전체 파이프라인 (7단계)](docs/figures/fig3_1_pipeline.png)

단계 (2)~(3)은 **가중치 결정** 이며, 본 논문의 연구 대상이다. 단계 (4)~(6)은 가중치에 따라 결정되는 **검색-융합-생성** 루프이다. LLM은 gpt-4o-mini (temperature=0.0, max_tokens=500) 으로 통일한다.

## 2절 세 지식 소스

### 2.1 Vector 소스 (α)
**정의.** 문서 코퍼스의 OpenAI text-embedding-3-small (1536-dim) 임베딩을 FAISS IndexFlatIP (L2 정규화 후 내적 = 코사인 유사도)로 저장하고, 질의 임베딩의 top-k (본 연구 k=3) 문서를 반환한다.

**구현.** `src/rag/vector_store.py`. Embedder를 주입 가능한 의존성으로 분리하여 단위 테스트에서는 결정적 hash embedder로 교체 (181개 test 중 8개가 vector store 대상).

**합성 대학 코퍼스에서의 규모.** 2,542 documents (577 프로프로필 + 60 학과 설명 + 1,505 과목 설명 + 400 프로젝트 설명). 모든 임베딩은 단일 pass 로 실행 (~10초, $0.005).

**강점.** 의미 유사도에 강하며, 특히 *simple* 질의 유형에서 안정적 recall.

**약점.** 구조 관계가 명시되지 않은 비정형 텍스트만 다루므로 multi-hop / 조건부 질의에서 한계.

### 2.2 Graph 소스 (β)
**정의.** 엔티티 (학과, 교수, 과목, 프로젝트) 와 관계 (소속, 담당, 협력, 참여, 개설) 를 NetworkX MultiDiGraph 로 구축하고, 질의 키워드로 시드 노드를 찾은 뒤 BFS (max_depth=3)로 관련 경로를 반환한다. 각 경로는 `"A --[관계]--> B"` 형태의 문자열로 직렬화된다.

**구현.** `src/rag/graph_store.py`. 합성 대학 그래프의 규모:
- Nodes: 2,542 (60 Department + 577 Professor + 1,505 Course + 400 Project)
- Edges: ≥ 3,000 (소속 577 + 개설 ≤ 1,505 + 담당 ≥ 1,700 + 협력 1,158 + 참여 ≥ 800)

**강점.** 관계를 명시적으로 표현하여 multi-hop 추론 지원.

**약점.** 그래프 탐색은 시드 매칭에 의존하여 질의의 자연어 표현과 노드 명칭 간 불일치 시 fallback 필요.

### 2.3 Ontology 소스 (γ)
**정의.** OWL 2 클래스 계층 (Person → Professor → FullProfessor/AdjunctProfessor) 과 속성 (hasAge, belongsTo, teaches), 수치 제약 (age ≤ threshold) 을 Owlready2 로 표현. HermiT 추론기 가용 시 결정 가능한 추론을 수행하며, 미가용 시 규칙 기반 fallback으로 동일 결과 제공.

**구현.** `src/rag/ontology_store.py`. `check_constraint(entity, phrase)` 가 "40세 이하", "30세 이상" 등 한국어 수치 제약을 정규식 파싱하여 대상 엔티티의 age 속성과 대조.

**강점.** 조건부 질의 (conditional query) 의 수치·논리 제약을 엄격히 처리.

**약점.** 코퍼스 초기 구축 비용이 크며, 개체 집합의 범위 변화에 대응하려면 수동 또는 자동 재구축 필요.

## 3절 QueryAnalyzer

**정의.** 자연어 질의에서 개체(entities), 관계(relations), 제약(constraints) 를 정규식·키워드로 추출하고, 각 density 신호 s_e, s_r, s_c ∈ [0, 1] 와 질의 유형(simple/multi_hop/conditional) 을 출력.

**출력 구조.**
```python
@dataclass
class QueryIntent:
    query_type: Literal["simple", "multi_hop", "conditional"]
    entities: list[str]
    relations: list[str]
    constraints: list[str]
    complexity_score: float
    density: tuple[float, float, float]  # (s_e, s_r, s_c)
```

**유형 분류 규칙.**
- `constraints` 비어있지 않으면 → `conditional`
- 엔티티 2개 이상 또는 관계 2개 이상 → `multi_hop`
- 그 외 → `simple`

---

> **[박스 3-1] 밀도 신호 (Density Signals) — 정의 · 예시 · 해석**
>
> **정의.** 규칙 기반 질의 분석기가 각 질의에서 추출하는 세 정수 신호 (*c<sub>e</sub>*, *c<sub>r</sub>*, *c<sub>c</sub>*). 각각 질의 내 개체(entity), 관계(relation), 제약(constraint) 토큰의 빈도.
>
> **형식 정의.**
> *c<sub>e</sub>* = |{e ∈ q : e ∈ 𝓔}|,  *c<sub>r</sub>* = |{r ∈ q : r ∈ 𝓡}|,  *c<sub>c</sub>* = |{t ∈ q : t ∈ 𝓒}|
>
> 여기서 𝓔 는 학과·교수·과목 엔티티 사전, 𝓡 는 담당·소속 같은 관계 술어, 𝓒 는 "N세 미만" 등 수치 제약 토큰의 집합.
>
> **예시.** 질의 *"기계공학과 소속 55세 이하 교수는?"*:
>
> | 신호 | 추출 토큰 | 값 |
> |---|---|---|
> | *c<sub>e</sub>* | "기계공학과", "교수" | 2 |
> | *c<sub>r</sub>* | "소속" | 1 |
> | *c<sub>c</sub>* | "55세 이하" | 1 |
>
> **해석.** (*c<sub>e</sub>*, *c<sub>r</sub>*, *c<sub>c</sub>*) = (2, 1, 1). R-DWA는 이 신호를 보고 "제약이 하나라도 있으니 Ontology 가중치(γ)를 높이자"고 판단 (Ch.Ⅳ). L-DWA는 동일 신호를 18-dim state에 포함하여 신경망으로 가중치를 학습 (Ch.Ⅴ).

---

**향후 확장.** §Ⅴ.2의 BERT multi-label intent classifier (`src/intent/bert_classifier.py`) 는 multi-label 확률 분포를 제공하여 L-DWA의 state (intent_logits 3-dim) 를 풍부화할 수 있다. 본 논문의 실험에서는 미학습 상태로 intent_logits=(0,0,0) 으로 둔다 (Ch.6 §5.2 참조).

## 4절 Score Fusion: merge_contexts

세 소스의 결과를 가중치 비례로 슬롯 할당하여 하나의 context 문자열로 조합한다. `top_k=3`, `budget = top_k × 3 = 9` 슬롯 기준:

```python
n_v = max(1, round(9 · α))
n_g = max(1, round(9 · β))
n_o = max(1, round(9 · γ))

context = [
    f"[Vector(α={α:.2f})]\n" + "\n".join(V[:n_v]),
    f"[Graph(β={β:.2f})]\n"  + "\n".join(G[:n_g]),
    f"[Ontology(γ={γ:.2f})]\n" + "\n".join(O[:n_o]),
]
```

**설계 동기.** 가중치가 실제 LLM 입력의 **슬롯 분배** 에 반영되어야 답변 품질이 변화한다. 본 구현은 각 소스에 최소 1 슬롯을 보장 (max(1, ·)) 하되, 큰 가중치의 소스가 더 많은 슬롯을 획득하도록 비례 배분한다.

**구현.** `src/rag/triple_hybrid_rag.py::merge_contexts` (모듈 레벨 함수, Ch.6의 offline cache builder와 evaluate_rerun에서 공통 사용).

---

> **[박스 3-2] 가중치 단순체 Δ³ — 정의 · 예시 · 해석**
>
> **정의.** 3개 소스 가중치 벡터 (α, β, γ) 가 만족해야 하는 제약 공간. "모든 비중은 0 이상, 합은 정확히 1".
>
> **형식 정의.**
> Δ³ = {(α, β, γ) ∈ ℝ³ :  α + β + γ = 1,  α, β, γ ≥ 0}
>
> **예시.** Δ³ 위 주요 세 점:
>
> | 조합 | (α, β, γ) | 의미 |
> |---|---|---|
> | Vector-only | (1.0, 0.0, 0.0) | 임베딩만 사용, Graph/Ontology 무시 |
> | Uniform | (0.33, 0.33, 0.33) | 세 소스 동등 가중치 |
> | Ontology-heavy | (0.1, 0.2, 0.7) | 조건부 질의에 온톨로지 우선 |
>
> **해석.** Δ³ 위의 한 점이 "어느 소스를 얼마나 믿을지" 에 대한 **하나의 결정**. R-DWA는 **규칙** 으로, L-DWA는 **학습** 으로 이 점을 질의마다 선택한다. Offline cache는 Δ³ 위의 **66개 이산 격자점** (10-grid 단순체 이산화) 만을 저장하여 PPO 학습의 계산 비용을 85% 절감한다 (§Ⅵ.10).

## 5절 Prompt Template

모든 실험은 다음 한국어 지시 프롬프트로 통일되어, 프롬프트 튜닝이 실험 간 교란 변수로 작용하지 않도록 고정한다:

```
다음 컨텍스트를 기반으로 질문에 정확하게 답변하세요.
컨텍스트에서 답을 찾을 수 없는 경우에만 '정보를 찾을 수 없습니다'라고 답하세요.
Graph 경로 정보(A --[관계]--> B)도 참고하여 관계를 추론하세요.
가능한 한 구체적이고 상세하게 답변하세요.

Context:
{context}

Question: {query}

Answer:
```

**주요 선택.**
1. "가능한 한 구체적이고 상세하게" — 짧은 답변 회피, F1 기여 증가
2. "Graph 경로 정보 참고" — LLM이 관계 기호를 자연어로 재해석하도록 가이드
3. 모호 시 "정보를 찾을 수 없습니다" — 환각 억제

## 6절 Pluggable DWA Interface

R-DWA (Ch.4) 와 L-DWA (Ch.5) 는 모두 다음 추상 인터페이스를 구현한다:

```python
class BaseDWA(ABC):
    @abstractmethod
    def compute(self, query: str, intent: QueryIntent) -> DWAWeights:
        """(α, β, γ) on the 3-simplex."""
```

`DWAWeights` 는 simplex 제약 (`α + β + γ = 1`, 각 ∈ [0, 1]) 을 `__post_init__` 에서 검증하는 frozen dataclass. 이 추상화 덕분에 `TripleHybridRAG.query()` 본체는 R-DWA/L-DWA/Uniform/Vector-only 등 **모든 정책에 대해 동일** 하다.

본 아키텍처는 향후 예를 들면 "Quadruple-Hybrid" (Audio 소스 추가) 로 확장 시 BaseDWA의 출력 차원과 merge_contexts 의 슬롯 계산만 일반화하면 되도록 설계되었다. 본 논문은 3-simplex에 국한한다.

## 본 장 핵심 기여의 위치

Ch.3은 본 논문의 **플랫폼** 이다. R-DWA (Ch.4) 와 L-DWA (Ch.5) 가 이 플랫폼 위의 대체 가능한 정책이며, Ch.6에서 두 정책을 동일 플랫폼에서 공정 비교한다. 플랫폼 자체 (세 소스의 정의, merge_contexts, Prompt) 는 선행 논문과 공유되어 재현성을 보장한다.
